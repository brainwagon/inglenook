# Inglenook Sidings — Functional Specification

## 1. Overview

Inglenook Sidings is a single-player 3D railway shunting puzzle. The player drives a locomotive across four tracks, coupling and decoupling freight cars to assemble a randomly generated five-car consist in exact order. The game is rendered in real-time 3D using Three.js and runs entirely in the browser with no server component.

---

## 2. Game Objective

A **target consist** of 5 cars is randomly selected (without replacement) from a pool of 8 and displayed to the player. The player wins by coupling exactly those 5 cars to the locomotive in the specified left-to-right order.

---

## 3. Game State

The authoritative game state contains:

| Field | Type | Description |
|-------|------|-------------|
| `sidings` | `{ A: number[], B: number[], C: number[], D: number[] }` | Car IDs present on each track, ordered throat-first |
| `locoTrack` | `string` | Track the locomotive currently occupies (`'A'`–`'D'`) |
| `coupled` | `number[]` | Car IDs coupled to the locomotive, in coupling order |
| `target` | `number[]` | The 5-car target consist (fixed for the duration of a game) |
| `moves` | `number` | Count of completed moves |
| `won` | `boolean` | Whether the win condition has been met |

### 3.1 Initial State

| Track | Contents |
|-------|----------|
| A (headshunt) | Empty — locomotive starts here |
| B | Cars 1, 2, 3, 4, 5 |
| C | Cars 6, 7, 8 |
| D | Empty |

The locomotive starts on track A with no coupled cars. The target is generated by shuffling all 8 car IDs and taking the first 5.

---

## 4. Track Layout

### 4.1 Geometry

The layout uses a two-level switch hierarchy with a constant switch angle of **15°** (π/12 radians).

**Switch Point 1 (SP1)** at coordinates (1, 0, 0):
- Straight leg: tracks A and B (colinear, at z = 0)
- Diverging leg: 15° diagonal toward SP2

**Switch Point 2 (SP2)** at approximately (1 + 2/tan 15°, 0, 2):
- Straight leg: continues the 15° diagonal to track D
- Diverging leg: bends back to horizontal for track C

A **switch lead** of 1 unit separates each switch point from the nearest track throat.

### 4.2 Track Definitions

| Track | Throat (x, z) | Direction | Length | Capacity | Notes |
|-------|---------------|-----------|--------|----------|-------|
| A | (0, 0) | Left (−1, 0) | 14 | 3 | Headshunt; loco sits at far-left end |
| B | (2, 0) | Right (+1, 0) | 14 | 5 | Colinear with A |
| C | (SP2.x+1, SP2.z) | Right (+1, 0) | 10 | 3 | Parallel to B |
| D | (SP2.x+cos15°, SP2.z+sin15°) | (+cos15°, +sin15°) | 10 | 3 | Continues the diagonal |

Tracks B, C, and D have buffer stops at their far ends.

### 4.3 Fill Direction

Each track has a **fill direction** — the unit vector along which successive cars are placed starting from the locomotive:

- **Track A:** Rightward (+1, 0, 0) — reversed from the track direction. The loco occupies the far-left end; cars fill back toward the throat.
- **Tracks B, C, D:** Same as the track direction. The loco sits at the throat; cars fill toward the buffer.

### 4.4 Routing

All moves pass through one or both switch points. Valid routes:

| From → To | Waypoints |
|-----------|-----------|
| A ↔ B | A.throat → SP1 → B.throat |
| A ↔ C | A.throat → SP1 → SP2 → C.throat |
| A ↔ D | A.throat → SP1 → SP2 → D.throat |
| B ↔ C | B.throat → SP1 → SP2 → C.throat |
| B ↔ D | B.throat → SP1 → SP2 → D.throat |
| C ↔ D | C.throat → SP2 → D.throat |

In practice, only A ↔ {B, C, D} moves are allowed by the game rules (see §5.1).

## 5. Game Rules

### 5.1 Movement

The locomotive can move between tracks subject to these constraints:

1. **Cannot move to the current track.**
2. **From the headshunt (A):** the loco may move to any siding (B, C, or D). Always allowed — no capacity check on arrival.
3. **From a siding (B/C/D):** the loco may only return to A. The move is allowed only if the number of cars remaining on the departing siding does not exceed that siding's capacity.

Each completed move increments the move counter by 1.

### 5.2 Coupling

The player may couple one car at a time from the siding the locomotive currently occupies.

- The car **nearest the locomotive** (throat end of the siding) is taken.
  - On B/C/D: the first element of the siding array (`.shift()`).
  - On A: the last element of the siding array (`.pop()`).
- The car is appended to the `coupled` array.
- **Limit:** a maximum of 5 cars may be coupled (equal to `targetSize`).
- **Blocked if:** the siding is empty, or the coupled array is already full.

After each coupling action, the win condition is checked immediately.

### 5.3 Decoupling

The player may decouple **all** coupled cars at once onto the current siding.

- **Not allowed on track A** (the headshunt).
- Coupled cars are placed at the throat end of the siding:
  - On B/C/D: `siding.unshift(...coupled)`.
- The `coupled` array is cleared.
- Decoupling is always allowed regardless of siding capacity (capacity is enforced when leaving).

### 5.4 Win Condition

After each coupling action, the game checks:

1. `coupled.length === 5`
2. For every index `i` from 0 to 4: `coupled[i] === target[i]`

If both conditions hold, the game is won. The `won` flag is set, a victory overlay is displayed, and a drive-off animation plays.

## 7. Positioning

### 7.1 Locomotive Stop Position

On **track A**, the locomotive always occupies a fixed position 1.2 units from the far-left end of the track, regardless of coupled or siding cars.

On **tracks B, C, D**, the locomotive stop position is computed dynamically so that the rear of the consist (locomotive + coupled cars) is separated from the nearest siding car (or buffer stop) by one inter-car gap:

1. **Contact distance** = distance from the throat to the near edge of the leftmost siding car (or to the buffer if the siding is empty).
2. **Rear extent** = distance from the loco center to the far edge of the last coupled car (or to the loco's own far edge if nothing is coupled).
3. **Loco distance from throat** = max(1.2, contact distance − rear extent − inter-car gap).

### 7.2 Coupled Car Positioning

Coupled cars are placed at successive multiples of slot spacing (2.2 units) from the locomotive center, in the fill direction of the current track.

### 7.3 Siding Car Positioning

On B, C, and D, siding cars are **buffer-packed**: the rightmost car's center is 0.8 units from the buffer, with each preceding car spaced 2.2 units toward the throat.

On A, siding cars fill from the far-left end toward the throat, offset to make room for the locomotive and any coupled cars.

## 8. Animation

### 8.1 Movement Animation

When a move is executed:

1. A polyline path is constructed from waypoints: loco position → route through switch points → destination stop position.
2. The path is extended beyond both endpoints so trailing cars always have valid interpolation positions.
3. The locomotive follows the path over time using smooth-step easing: `t² × (3 − 2t)`.
4. Coupled cars are positioned along the path at fixed distance offsets from the locomotive:
   - Moving right (toward buffer): cars are **ahead** of the loco on the path.
   - Moving left (toward throat/A): cars are **behind** the loco on the path.
5. Rotation is derived from the path tangent at each entity's position, with a **no-flip guard** that prevents 180° orientation reversals by choosing the closer of `angle` or `angle ± π` relative to each mesh's starting rotation.

**Duration:** (number of path segments) × 1.5 seconds per segment.

**Interaction lock:** all input is disabled while an animation is playing.

### 8.2 Victory Animation

On win, the locomotive and all coupled cars drive off-screen to the left (x = −30) along track A. If the loco is not already on A, it routes there first. The animation uses the same polyline system.

## 9. User Interface

### 9.1 HUD Elements

| Element | Position | Content |
|---------|----------|---------|
| Target panel | Top-left | "Target Consist" heading + 5 colored/numbered car indicators |
| Info panel | Top-right | Move counter; status line showing loco location, coupled cars, and cars on the current siding |
| Move buttons | Bottom-center | "Move to:" label + A, B, C, D buttons |
| Action buttons | Bottom-center | Couple (green), Decouple (red) |
| New Game button | Bottom-center | Resets the game |
| Error toast | Bottom-center | Auto-dismissing (1.5 s) red notification for invalid actions |
| Victory overlay | Full-screen | "Victory!" title, move count, "Play Again" button |

### 9.2 Button States

Move buttons are **disabled** when the destination is invalid:
- The button for the loco's current track is always disabled (highlighted gold).
- When the loco is on a siding, all buttons except A are disabled.

All buttons are disabled during animations and after victory.

### 9.3 Mouse Interaction

- **Clicking a track** (via invisible click zones) moves the loco there if valid.
- **Clicking a car or the loco** resolves to a track click for the entity's track.
- **Hovering** over a car or loco adds an emissive highlight (0x333333) and changes the cursor to `pointer`.
- **OrbitControls** allow rotating and zooming the camera (polar angle clamped between 0.3 and π/2.2 radians; distance clamped between 8 and 50 units).

## 10. 3D Scene

### 10.1 Renderer

- WebGL with antialiasing, soft shadow mapping (PCFSoftShadowMap), 2048×2048 shadow map.
- Background color: #1A1A2E.
- Pixel ratio: min(devicePixelRatio, 2).

### 10.2 Camera

- Perspective, 45° FOV.
- Initial position: (2, 30, 22), looking at (4, 0, 1.5).

### 10.3 Lighting

- **Ambient light:** white, intensity 0.5.
- **Directional light:** white, intensity 0.8, position (10, 20, 10), casts shadows.

### 10.4 Ground

- 60 × 60 unit gray plane (#4A4A4A) at y = −0.05.

### 10.5 Track Visuals

- Two parallel steel rails (gray, metallic) per segment, 0.3 units apart.
- Wooden sleepers every ~1.2 units.
- Red buffer stops with diagonal braces at the ends of B, C, D.
- Letter labels (A–D) as sprites above each track's far end.

### 10.6 Scenery

~26 decorations (boulders and pine trees) are placed around the track perimeter using a seeded pseudo-random generator (seed 12345) for deterministic placement across reloads. Approximately 45% boulders, 55% pine trees. Each tree is a composite of a cylinder trunk and three stacked cone tiers with flat shading.

## 11. Module Architecture

The application is composed of eight modules, each an IIFE exporting a public API:

| Module | File | Responsibility |
|--------|------|----------------|
| CONFIG / LANG | config.js | Constants, dimensions, colors, strings |
| Tracks | tracks.js | Track geometry, switch points, routing, entity positioning |
| SceneManager | scene.js | Three.js renderer, camera, lighting, ground, render loop |
| Entities | entities.js | Creates and positions all 3D meshes (loco, cars, rails, scenery) |
| GameState | gameState.js | Game logic: state management, coupling, decoupling, move validation, win check |
| HUD | hud.js | DOM updates: status, target display, victory overlay, error toasts |
| Animation | animation.js | Polyline path-following animation for moves and victory drive-off |
| Interaction | interaction.js | Raycasting, click/hover handlers, button wiring |

**Bootstrap** (`main.js`) initializes modules in dependency order and starts the render loop.
